Lecture 16: Méthode PUT

Développement d’API REST : Construire, Tester, Dépanner

La méthode PUT est utilisée dans les API REST pour mettre à jour une ressource existante. Contrairement à la méthode POST, qui crée de nouvelles données, PUT modifie les données qui existent déjà sur le serveur.

Les clients envoient des informations mises à jour à l’API, et le serveur remplace ou met à jour la ressource existante avec les nouvelles valeurs fournies dans la requête.

Objectif de la méthode PUT

PUT est utilisé lorsqu’un client doit mettre à jour une ressource spécifique. La requête inclut généralement l’identifiant de la ressource ainsi que les nouvelles données.

Exemples courants :

  • Mettre à jour les informations d’un produit
  • Modifier les détails d’un client
  • Changer le statut d’une commande
  • Modifier les informations du profil utilisateur

Exemple de requête PUT

Une requête PUT est généralement envoyée à un point d’accès spécifique qui inclut l’identifiant de la ressource à mettre à jour.

PUT /api/products/5

Corps de la requête :

{ "id": 5, "name": "Wireless Mouse", "price": 22, "stock": 60 }

Cette requête indique au serveur de mettre à jour le produit avec l’ID 5 en utilisant les nouvelles valeurs fournies.

Traitement côté serveur

Lorsque le serveur reçoit une requête PUT, il effectue généralement les étapes suivantes :

  • Vérifier que la ressource existe
  • Valider les données reçues
  • Mettre à jour l’enregistrement dans la base de données
  • Retourner une réponse au client

Exemple de réponse

Si la mise à jour est réussie, le serveur peut retourner la ressource mise à jour ou un message de confirmation.

{ "id": 5, "name": "Wireless Mouse", "price": 22, "stock": 60 }

Codes de statut HTTP pour PUT

  • 200 OK – Ressource mise à jour avec succès
  • 204 No Content – Mise à jour réussie sans contenu de réponse
  • 400 Bad Request – Données invalides
  • 404 Not Found – Ressource introuvable

Tester les requêtes PUT

Les développeurs peuvent tester les requêtes PUT à l’aide d’outils tels que Postman. Ces outils permettent d’envoyer des données JSON dans le corps de la requête et de vérifier la réponse de l’API.

Exemple d’endpoint testé dans Postman :

https://localhost:5001/api/products/5

Bonnes pratiques

  • Toujours valider les données entrantes
  • S’assurer que la ressource existe avant la mise à jour
  • Retourner des codes de statut HTTP appropriés
  • Utiliser des structures JSON claires et cohérentes

Le respect de ces pratiques permet de maintenir une API REST fiable et prévisible.

Étapes suivantes

Après avoir implémenté la méthode PUT, l’étape suivante consiste à implémenter la méthode DELETE, qui permet de supprimer des ressources et de compléter les fonctionnalités CRUD complètes au sein de l’API.
